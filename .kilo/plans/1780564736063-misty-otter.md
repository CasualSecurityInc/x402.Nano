# Rev 8 Two-Track Reorganization

## Background

Rev 8 currently has only `nanoSignature` (NOMS-based post-payment proof). The user wants two tracks:

- **Track A: `nanoTxn`** — naive, frontier-dependent signed-block approach (like EVM/SVM: client signs, facilitator broadcasts)
- **Track B: `nanoSignature`** — existing NOMS-based post-payment proof (client sends on-chain first, then proves ownership)

## Research Summary

### Source Material for Track A (synthesized from `docs/references/`)

**`schemes/exact.md`** (`@x402nano/exact`) — Primary reference for signed-block pattern:
- Client constructs full Nano state send block (account, previous, representative, balance, link, signature, work)
- PaymentPayload contains the full block as `payload.block`
- Facilitator verifies: balance check, amount decrement match (previousBalance − newBalance = amount), destination match (link decodes to payTo), block signature verification, work format validation
- Facilitator settles by broadcasting via `process` RPC and waiting for confirmation
- Fork protection: track `block.previous` to prevent duplicate frontier submissions

**`x402Nano-API/API_DOCUMENTATION.md`** — Useful patterns:
- Unique amount fingerprint: server adds a micro-offset to requested amount to make it unique traceable on-chain
- Transaction expiry (60 minutes)
- `/transaction/create` → `/transaction/pay` → `/transaction/status` flow model

**`@x402nano/helper`** (`helper/README.md`):
- `generateSendBlock({ payTo, amount })` — generates and signs blocks
- `processBlock({ block })` — publishes block via RPC
- `getAccountInfo({ account })` — gets current account state

**`x402-foundation/specs/scheme_impl_template.md`** — Template structure:
- Summary, Payment header payload, Verification, Settlement, Appendix

### 2. Create `x402_NanoSession_rev8_Track_A_nanoTxn.md`

Synthesize from references into a clean spec following Rev 7's formatting style:

**Sections:**

1. **Summary** — Post the naive signed-block approach. Client signs a Nano send block, facilitator validates and broadcasts it. Known compromise: Nano's frontier dilemma (unrelated account activity before broadcast invalidates the block).

2. **Network Identifier** — `nano:mainnet`

3. **Asset** — `XNO`, amounts in raw, base-10 integer string

4. **Roles** — Client (constructs + signs block), Resource Server (passes headers), Facilitator (verifies + broadcasts block)

5. **PaymentRequirements** — Standard x402 v2 `accepts` entry. Extra fields:
   - `extra.tagAmountRaw`: challenge-unique raw fingerprint (8 bytes random, hex-encoded as 16 chars)
   - `extra.expiresAt`: ISO-8601 challenge expiry

6. **Block Construction** — Client constructs a Nano state send block:
   - Fetches account frontier via `account_info` RPC
   - `link` = decoded payTo address bytes
   - `balance` = current balance − amount
   - Signs with account private key
   - Generates work (PoW)
   - amount = `accepted.amount` which is `resourceAmountRaw + tagAmountRaw`

7. **PaymentPayload** — Full signed block in `payload.block`:
   ```json
   {
     "block": {
       "type": "state",
       "account": "nano_1sender...",
       "previous": "F47B...",
       "representative": "nano_1rep...",
       "balance": "999000000...",
       "link": "19D3D9...",
       "link_as_account": "nano_3recv...",
       "signature": "3BFBA6...",
       "work": "ffffffd2e1234567"
     }
   }
   ```

8. **Facilitator Verification** — Ordered checks (reference `schemes/exact.md`):
   1. Parse payload structure, validate fields
   2. Check expiry (`expiresAt`)
   3. Verify block type = "state" and link decodes to payTo
   4. Verify amount: `previousBalance − newBalance = accepted.amount` (including tag)
   5. Verify `tagAmountRaw` matches issued challenge
   6. Fetch `account_info` for sender to verify current frontier = `block.previous`
   7. Verify block signature using account's Ed25519 public key
   8. Verify work field format (don't regenerate, just validate)
   9. Check fork protection: `block.previous` not in active verification set
   10. Record `block.previous` in fork-protection set

9. **Facilitator Settlement** — Broadcast block via `process` RPC, wait for confirmation

10. **Replay Protection** — Fork-protection via `block.previous` tracking + block hash seen-set after broadcast

11. **Fork Protection** — Track `block.previous` values during verification. Reject any new verification with the same `block.previous`. Clear after settlement or expiry.

12. **Security Considerations** — 
    - Frontier dilemma: any unrelated account activity before broadcast invalidates the block
    - Block-hash uniqueness after broadcast prevents replay
    - Amount tagging prevents cross-challenge borrowing on public chain
    - Tagged amounts make each challenge's on-chain payment unique
    - Best for single-purpose/agent wallets where frontier is controlled

13. **Error Codes** — Standard error codes (see `schemes/exact.md`)
